# Hanagashi Campus Story 开发手册

欢迎接手本项目。这里是一个基于 **WebGAL** 的视觉小说工程，重点围绕剧情脚本、演出编排、资源管理和可维护的章节推进展开。先读这份文件，再决定从哪个自定义 Agent 进入任务，能少走很多弯路。

## 这是一个什么项目

- 游戏类型：视觉小说 / Galgame
- 核心引擎：WebGAL
- 游戏根目录：`game/`
- 启动入口：`game/scene/start.txt`
- 全局配置：`game/config.txt`
- 主要工作内容：剧情章节、分支设计、演出脚本、立绘/背景/BGM/语音等资源落地

## 目录怎么理解

### `game/scene/`
放所有可执行剧情脚本。这里是最常改的地方。

### `game/background/`
背景图、标题图、宣传图等。通常配合 `changeBg`、标题配置或发布物料使用。

### `game/figure/`
角色立绘。WebGAL 立绘通常需要透明底 PNG，路径和文件名尽量稳定。

### `game/bgm/`
背景音乐文件。修改脚本前，先确认音频文件真实存在且命名一致。

### `game/vocal/`
语音文件。

### `game/animation/`
动画描述文件。比如入场、离场、抖动、闪烁、特效动画等。

### `game/tex/`
舞台特效纹理。

### `game/template/`
模板与通用样式资源。优先复用这里已有内容，避免重复造轮子。

### `docs/`
剧情文稿、章节规划、分支设计、设定说明等文字内容。这里偏“写作与策划”，不直接替代可执行脚本。

## 自定义 Agent 的分工

项目里已经有两个核心自定义 Agent：`StoryMaster` 和 `GalMaker`。

### `StoryMaster`
负责剧情层面的内容：

- 新章节写作
- 情节重写
- 分支设计
- 角色弧光整理
- 设定统一与连续性维护
- `docs/` 下的剧情文稿整理

它的工作方式是先做计划，再做正文；当剧情方案确认后，会把内容交给 `GalMaker` 落地成 WebGAL 脚本。

### `GalMaker`
负责工程落地层面的内容：

- WebGAL 脚本实现
- 场景演出调优
- 分支条件、变量门、回收点实现
- 资源路径排障
- `game/` 下的脚本和素材联动
- 主界面图、背景图、立绘、宣传图相关落库与使用

### 任务路由原则

- **写剧情、补章节、改分支、调叙事**：先用 `StoryMaster`。
- **把已确认剧情转成可执行脚本**：由 `StoryMaster` 交给 `GalMaker`，或直接找 `GalMaker`。
- **改 WebGAL 实现、资源路径、演出、配置**：直接用 `GalMaker`。
- **不确定该找谁时**：先看任务是“文字方案”还是“脚本实现”。文字方案归 `StoryMaster`，实现归 `GalMaker`。

## 工作流建议

1. 先判断需求属于剧情、脚本、资源，还是配置。
2. 需要剧情方案时，优先交给 `StoryMaster`。
3. 需要脚本实现、资源检查、路径排障时，优先交给 `GalMaker`。
4. 对于剧情任务，先确认分支拓扑、变量变化、回收点，再写脚本。
5. 对于实现任务，先确认资源文件是否存在，再改脚本引用。
6. 改完后至少做一次整体通读，检查路径、命令、变量名是否一致。

## WebGAL 关键约定

- `start.txt` 是入口，不要随意改成别的启动链路。
- WebGAL 按行解析；大多数命令以分号结尾，分号后是注释。
- 命令参数前通常要带空格，例如 `command:content -flag`。
- `changeScene`、`callScene`、`choose` 这类流程切换通常会继承已有舞台状态，要特别注意背景、立绘和 BGM 是否需要先清理。
- `none` 常用于关闭背景、立绘、头像、BGM 等资源。
- 条件执行通常用 `-when=...`。
- 需要同步执行的演出可用 `-next`，但不要滥用到让脚本难以读懂。

## 资源与命名注意点

- 文件名尽量使用英文、小写、无空格。
- 路径大小写要和真实文件完全一致。
- 立绘优先用透明底 PNG。
- 背景图尽量符合 16:9 比例。
- 同类素材如果要迭代，建议保留版本号，等稳定后再切换正式文件名。
- 先复用现有资源，再考虑新增；不要为了“看起来完整”而批量制造无用素材。

## 修改前先看什么

- 改剧情：先看 `docs/` 里的文稿、章节索引、角色关系与分支设计。
- 改脚本：先看 `game/scene/start.txt` 和目标章节脚本。
- 改配置：先看 `game/config.txt`，确认标题图、语言、包名、鉴赏开关等设置。
- 改素材：先确认目录中是否已有可复用资源，避免重复命名。

## 容易踩坑的地方

- 把剧情文稿直接当成可执行脚本。
- 改了脚本但资源文件名没同步。
- 分支设计只有“看起来分了”，但实际没有产生关系值、信息量或状态变化。
- 场景切换后忘了清理旧立绘、旧 BGM 或旧字幕框状态。
- 同时维护多套剧情规范，导致设定漂移。

## 剧情类任务的额外规则

如果任务涉及 `docs/` 中的剧情写作或修订：

- 先由 `StoryMaster` 做计划和文稿。
- 若需要形成长期有效的剧情规范，只维护一份统一来源，不要拆成多份同级规则文件。
- 章节文稿最好明确：选择点、分支后果、变量变化、汇合点、脚本落地锚点。
- 文字方案确认后，再交给 `GalMaker` 转脚本。

## 你在这里的默认目标

不是只“把功能做出来”，而是做到：

- 剧情可读
- 分支可走
- 资源可用
- 演出稳定
- 后续接手的人能看懂

如果你准备开始动手，先确认需求属于哪一类，再选对应 Agent。
